# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this plugin does

OpenClaw plugin that lets the AI agent control Home Assistant devices, schedule future automations via OpenClaw's cron system, and report back to the user via a configurable notification channel.

## Architecture

```
src/
  index.ts             Plugin entrypoint — registers tools, hooks, service, and CLI with OpenClaw
  ha-client.ts         Thin fetch wrapper for the HA REST API (/api/states, /api/services/*)
  device-registry.ts   Loads ~/.openclaw/workspace/ha/*.yaml into memory; builds context block
  setup.ts             Auto-generates workspace YAML files by fetching entities from HA
  tools/
    ha-get-states.ts   Tool: get current state of one or more entities
    ha-list-entities.ts Tool: list curated enabled devices with live state
    ha-call-service.ts  Tool: call a HA service + optional channel notification
    ha-schedule-action.ts Tool: validate + prepare cron.add parameters for a future action
skills/
  ha-control/SKILL.md  Instructs the agent how to resolve device names and call services
```

### Key design decisions

- **Device registry** (`~/.openclaw/workspace/ha/*.yaml`): one YAML file per domain, auto-generated by `openclaw ha init`. Users set friendly names, rooms, and `enabled: false` to exclude devices.
- **Ambient context injection**: a `command:new` hook injects all enabled devices (name + entity_id + room) into every agent turn so the LLM can resolve natural language references without a tool call.
- **Natural language matching is done by the LLM**, not by code — the device list in context lets the model map "donut lamp" to `light.donut_lamp` semantically.
- **Scheduling** uses OpenClaw's native `cron.add` agent tool. `ha_schedule_action` validates the device and returns structured `cron_add_params`; the agent then calls `cron.add` itself.
- **Reporting** calls `api.runtime.channels.send(channelId, message)` after `ha_call_service` succeeds. The channel is configured once via `reportingChannel` in the plugin config.

## Plugin configuration (openclaw.json)

```json
{
  "plugins": {
    "entries": {
      "ha": {
        "enabled": true,
        "config": {
          "haUrl": "http://homeassistant.local:8123",
          "haToken": "<long-lived-access-token>",
          "reportingChannel": "telegram",
          "reportingEnabled": true
        }
      }
    }
  }
}
```

## Workspace files

Device context lives in `~/.openclaw/workspace/ha/`. Generate them with:

```
openclaw ha init
```

Reload after editing YAML files without restarting OpenClaw:

```
openclaw ha reload
```

## HA REST API surface used

- `GET /api/states` — fetch all entity states (used in setup + ha_get_states)
- `GET /api/states/<entity_id>` — single entity state
- `POST /api/services/<domain>/<service>` — call a service
- `POST /api/template` — render a template (used in setup to attempt area resolution)

## Adding new tools

1. Create `src/tools/ha-<name>.ts` following the pattern in existing tools (return an object with `name`, `description`, `parameters`, `handler`)
2. Import and register with `api.registerTool?.(make<Name>Tool(...))` in `src/index.ts`
3. Update `skills/ha-control/SKILL.md` if the agent needs guidance on when/how to use it
